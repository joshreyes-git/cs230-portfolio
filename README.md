# cs230-portfolio

1) Briefly summarize The Gaming Room client and their software requirements

The Gaming Room was the client for this project, and they wanted a web-based game called Draw It or Lose It. Their main requirements included supporting multiple teams and players, keeping game data organized, and ensuring the game runs across different operating systems. They also needed the design to support scalability and future growth as more users join.

2) What did you do particularly well in developing this documentation?

I did well at organizing the design document in a clear, structured way so the requirements and decisions were easy to follow. I also explained the reasoning behind major design choices, such as how the system would manage users and game sessions. This helped connect the client’s needs to specific technical solutions.

3) What about the process of working through a design document did you find helpful when developing the code?

Writing the design document first helped me plan the system before jumping into coding. It made it easier to identify what classes and components were needed and how they should work together. Overall, it reduced confusion and made implementation more straightforward.

4) If you could revise one part of your work, what would you pick and how would you improve it?

If I could revise one part, I would expand the section that explains the system architecture and data flow. I would add more detail and possibly a simple diagram to show how requests move through the system. That would make the design easier to understand for someone new to the project.

5) How did you interpret the user’s needs and implement them into your software design, and why is it important?

I interpreted the user’s needs by translating the client’s requirements into clear features such as multi-team support, secure user management, and scalable deployment. It is important to consider the user’s needs because the design must solve the client’s real problems, not just be technically correct. A user-focused design also improves usability and reduces the chance of costly changes later.

6) How did you approach designing software, and what techniques would you use in the future?

I approached designing software by reviewing requirements first, then breaking the system into smaller parts like services, data models, and interfaces. In the future, I would continue using techniques like UML diagrams, user stories, and early prototyping to validate design decisions. I would also focus on designing for scalability and maintainability from the beginning.
